## Principles

Vanta's threat model ([21. Threat Model](21-threat-model.md)) names "compromised Vanta release" as total compromise, so Vanta holds itself to the same bar it imposes on tools ([15. Security](15-security.md)): every release is **reproducible, signed, provenance-attested, and verified before use**. The registry, the other root of trust, is signed with the same TUF-style roles it uses for tool metadata.

## Reproducible builds

- A **pinned toolchain** (`rust-toolchain.toml`) and pinned dependencies (`Cargo.lock`) fix the build inputs.
- Builds are **hermetic**: no network beyond the vendored/locked dependency set, normalized build environment (fixed paths via `--remap-path-prefix`, `SOURCE_DATE_EPOCH`, sorted inputs), so the same source yields **bit-identical binaries** independently.
- Reproducibility is verified in CI by building each target on two independent runners and comparing hashes; a mismatch blocks the release. This lets third parties (and the SLSA verifier) confirm the published binary matches the source.

## The build matrix

Vanta cross-compiles to every supported platform token ([17. Cross-platform](17-cross-platform.md)):

| Target | Notes |
| --- | --- |
| `linux/x86_64/gnu`, `linux/x86_64/musl` | musl is fully static (distro-agnostic, containers) |
| `linux/aarch64/gnu`, `linux/aarch64/musl` | ARM Linux |
| `macos/aarch64`, `macos/x86_64` | shipped also as a universal binary; codesigned + notarized |
| `windows/x86_64`, `windows/aarch64` | Authenticode-signed |

Both `vanta` and `vanta-shim` are built per target. The musl static builds make Vanta itself dependency-free on Linux, consistent with the single-static-binary goal ([ADR-0002](24-architecture-decision-records.md)).

## Signing, provenance, and SBOM

Every release artifact carries:

- **A signature** — cosign-signed (and a detached minisign signature), against the release keys ([key management](#key-management)).
- **SLSA provenance** — generated by the build pipeline attesting the source commit, builder, and inputs (verifiable, given reproducibility).
- **A CycloneDX SBOM** — the dependency inventory of the binary.
- **A checksums file** — SHA-256 + BLAKE3 for every artifact.
- Platform-native signing in addition: **codesigning + notarization** on macOS (Gatekeeper), **Authenticode** on Windows (SmartScreen) ([17. Cross-platform](17-cross-platform.md)).

`vanta self update` and every package channel verify the signature + provenance before trusting a binary.

## Release channels

| Channel | Source | Audience | Signing |
| --- | --- | --- | --- |
| `nightly` | every `main` commit | contributors, early testing | signed + attested dev builds |
| `beta` | pre-release stabilization | adopters validating an upcoming release | signed |
| `stable` | tagged releases | everyone (default) | signed, supported |

Promotion flows `nightly → beta → stable`; a stable tag is the artifact that distribution channels publish. `vanta self update --channel` selects the channel ([12. Updates](12-updates.md)).

## Distribution

All channels deliver the **same signed binary**; the wrapper differs:

```sh
# Shell installer (Linux/macOS) — verifies TLS + checksum + signature before install
curl --proto '=https' --tlsv1.2 -fsSL https://get.vanta.dev | sh

# Windows
irm https://get.vanta.dev/install.ps1 | iex
winget install vanta
scoop install vanta

# Package managers / build
brew install vanta-dev/tap/vanta
cargo binstall vanta          # prebuilt; cargo install vanta to build
```

`.deb`/`.rpm` packages and a container image (`ghcr.io/vanta-dev/vanta`) are produced for every stable tag. Each distribution path verifies the signature/checksum so a compromised mirror cannot substitute a binary. The installer is itself small and auditable, and pins the expected release public key.

## Verified self-update

`vanta self update` ([12. Updates](12-updates.md)) is the in-place updater and is fully verified and reversible:

```
1. query the selected channel for the newest release + its metadata
2. download the target binary, its signature, provenance, and checksums
3. verify: checksum, signature (against pinned release keys), provenance ≥ policy   (FAIL CLOSED)
4. stage the new binary; atomically rename over ~/.vanta/bin/vanta; keep the prior binary
5. run a post-update self-check; on failure, restore the prior binary (auto-rollback)
```

When Vanta was installed by a system package manager (brew/winget/apt), `self update` detects this and defers to that manager with a clear message rather than fighting it. `vanta self update --rollback` restores the retained prior binary.

## The registry signing pipeline

The official registry is the other root of trust and is built and signed deliberately ([26. Registry Reference](26-registry-and-metadata-reference.md), [15. Security](15-security.md)):

- **Index + provider builds** are generated from reviewed sources; a new tool/version or provider goes through review and the provider test harness ([22. Provider SDK](22-provider-sdk.md), [28. Testing](28-testing.md)) before inclusion.
- **Signing with TUF-style roles** — `registry`/`targets` signs the index; `snapshot` fixes a consistent `registry_revision`; `timestamp` bounds freshness; `root` (offline, threshold) delegates and rotates.
- **Community providers** are submitted, reviewed, tested, and signed into the index by a verified publisher identity; namespaces are reserved to prevent typosquatting.
- The registry is itself reproducible and auditable; a `registry_revision` is an immutable, signed snapshot a lock can pin ([11. Reproducibility](11-reproducibility.md)).

## Build dependency supply chain

The supply chain *into* Vanta's build is governed:

- `cargo-deny` (licenses/advisories/bans), `cargo-audit` (RustSec), and `cargo-vet` (trusted dependency review) run in CI; a new or vulnerable dependency blocks the build ([03. Repository](03-repository.md)).
- Dependencies are pinned (`Cargo.lock`) and can be vendored for hermetic/air-gapped builds.
- The `[workspace.dependencies]` discipline keeps versions single-sourced; dependency additions require review.

## Key management

- **Release keys** and **registry root keys** are kept **offline**, use **threshold (M-of-N)** signing for the root role, and have documented **rotation** procedures (a root-signed delegation update) so loss of a single key is recoverable without re-bootstrapping trust.
- Clients pin the root keys (shipped with Vanta, rotatable via the root role); compromise of a non-root role is bounded by the role model ([21. Threat Model](21-threat-model.md)).
- Signing happens in an isolated CI step with restricted access; provenance records the signer.
